## Project Overview

DakarEaseU is a student super-app for Dakar with two frontends sharing a single Supabase backend:
- `apps/admin` — Next.js 16 dashboard (App Router, deployed on Vercel)
- `apps/mobile` — Expo SDK 54 / React Native 0.81.5 mobile app (deployed via EAS)
- `packages/types` — auto-generated Supabase DB types + convenience type aliases
- `packages/shared` — Zod schemas, constants, and utilities shared across apps

This is an npm workspaces monorepo. All installs must be run from the root.

## Commands

### Root (all workspaces)
```bash
npm install          # install all workspace dependencies
npm run lint         # ESLint across all workspaces
npm run typecheck    # tsc --noEmit across all workspaces
npm run test         # run all tests
npm run build        # build all workspaces
npm run format       # Prettier write
npm run format:check # Prettier check (used in CI)
```

### Admin (Next.js) — run from `apps/admin/`
```bash
npm run dev          # start dev server at http://localhost:3000
npm run build        # production build
npm run test         # vitest run (single pass)
npm run test:watch   # vitest watch mode
npm run typecheck    # tsc --noEmit
```

### Mobile (Expo) — run from `apps/mobile/`
```bash
npx expo start       # start Expo dev server (scan QR or press i/a)
npm run test         # jest
npm run typecheck    # tsc --noEmit
```

### Supabase
```bash
npx supabase db push                    # apply migrations to remote project
npx supabase gen types typescript ...   # regenerate packages/types/src/database.types.ts
```

### Single test file
```bash
# Admin (vitest)
cd apps/admin && npx vitest run src/path/to/file.test.ts

# Mobile (jest)
cd apps/mobile && npx jest src/path/to/file.test.ts
```

## Architecture

### Shared Backend: Supabase
All data lives in Supabase (PostgreSQL + Auth + Storage + Realtime). There is no custom server. Row-Level Security (RLS) enforces permissions at the DB level. Migrations live in `supabase/migrations/` and are applied sequentially. The generated types in `packages/types/src/database.types.ts` must stay in sync with the actual schema.

### `packages/types`
Single source of truth for DB-derived types. `database.types.ts` is generated by the Supabase CLI. `index.ts` re-exports convenience aliases (`Profile`, `Listing`, `Booking`, etc.) and enum types used throughout both apps. Never hand-edit `database.types.ts`.

### `packages/shared`
Cross-app Zod schemas (e.g. `listing.schema.ts`) and constants (routes, enum labels). Import via `@dakareaseu/shared` and `@dakareaseu/types`.

### Admin App (`apps/admin`)
- **Next.js App Router** with route groups: `(auth)` for login, `(dashboard)` for all protected pages.
- **Middleware** (`middleware.ts`) handles auth gating — unauthenticated users are redirected to `/login`; authenticated users are redirected away from public paths.
- **Supabase clients**: three separate clients in `src/lib/supabase/`:
  - `browser-client.ts` — singleton for Client Components
  - `server-client.ts` — created per-request for Server Components / Route Handlers
  - `admin-client.ts` — uses `service_role` key for privileged operations
- **Feature structure**: `src/features/<domain>/{components,hooks,services,schemas}`. Each feature owns its own data-fetching hooks (React Query) and Supabase service functions.
- **UI components**: shadcn/ui in `src/components/ui/`. Shared layout components in `src/shared/components/`.
- **Providers**: `QueryClientProvider` + `Sonner` toaster wired in `src/app/layout.tsx`.
- **Testing**: Vitest + React Testing Library with jsdom. Aliases for `@dakareaseu/types` and `@dakareaseu/shared` are configured in `vitest.config.ts`.

> **Important**: Read `node_modules/next/dist/docs/` before writing Next.js code — this project uses Next.js 16 which may have breaking API changes from your training data.

### Mobile App (`apps/mobile`)
- **expo-router** file-based navigation. Route groups: `(auth)` for onboarding/login, `(tabs)` for the main tab bar.
- **Supabase client** in `src/lib/supabase.ts` uses AsyncStorage for session persistence (no cookie-based auth).
- **Feature structure**: mirrors admin — `src/features/<domain>/{screens,components,hooks,services,schemas,store}`.
- **State**: React Query for server state, Zustand stores for local UI state (`src/shared/store/`, `src/features/auth/store/`).
- **Providers**: `AppProviders` wraps `QueryClientProvider > SessionProvider > RealtimeProvider`.
- **i18n**: `src/lib/i18n/` with fr/en/wo (Wolof) locales. Access via `useTranslation` hook.
- **Styling**: NativeWind (Tailwind for React Native). Use `className` prop on RN components.
- **Testing**: Jest + jest-expo + React Native Testing Library.

## Environment Variables

| Prefix | Exposed to |
|--------|-----------|
| `NEXT_PUBLIC_` | Admin browser bundle |
| `EXPO_PUBLIC_` | Mobile Expo bundle |
| No prefix | Server-only (never in client bundle) |

`.env` at repo root is for server secrets (CI/CD, Supabase service role, Vercel, Expo tokens). Each app has its own `.env`/`.env.local` for public vars. Copy `.env.example` files — never commit real secrets.

## CI/CD

- **CI** (`.github/workflows/ci.yml`): lint → typecheck → test → build on PRs and pushes to `main`/`develop`.
- **Admin deploy** (`.github/workflows/deploy-admin.yml`): auto-deploys to Vercel on pushes touching `apps/admin/**`.
- **Mobile deploy** (`.github/workflows/deploy-mobile.yml`): manual trigger or git tag (`v*`) deploys via EAS Build.

Required GitHub Secrets: `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_DB_PASSWORD`, `VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`, `EXPO_TOKEN`.
